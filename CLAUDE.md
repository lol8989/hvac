## 1. 프로젝트 개요

LG전자 HVAC(공조) 도면을 다루는 **AgentOne Architect** 협업 프로젝트. 세 개의 핵심 도메인으로 구성된다.

| 도메인 | 설명 | 주체 |
|--------|------|------|
| **장비마스터 (Equipment Master)** | 장비 모델·스펙·단가의 단일 진실 공급원(SSOT). 등록/관리 전용. | 관리자 |
| **검도 (Review)** | 도면과 장비일람표의 정합성 검증. | 유저(협력사/영업) |
| **생성 (Generation)** | 도면 방 검출 → 실내기 배치 → 실외기 선정·조합 → 산출물(장비일람표·도면) 생성. | 유저 |

**핵심 원칙:** 장비마스터는 "등록만" 담당하고, 호환·조합 판단(조합비·계열 매칭)은 **유저(생성) 단**에서 마스터 데이터를 참조해 수행한다.

**현재 POC 범위:** React(Vite)로 만든 유저 페이지 프로토타입 — "생성 작업 · 방 검출 결과" 화면(SVG 도면 뷰어 + 우측 모델 선택 + 상단 조합 리포트 + AI 실내기 배치 + 실외기 조합 매핑 팝업).

## 2. 바운디드 컨텍스트 (DDD Context Map)

```
┌─────────────────────┐        참조(읽기)        ┌─────────────────────┐
│  Equipment Master    │ ───────────────────────▶ │  Generation          │
│  (장비마스터/관리자) │                          │  (생성/배치·조합)    │
│  SSOT: 모델·스펙·단가│ ───────────────────────▶ │                      │
└─────────────────────┘                          └─────────┬───────────┘
          │  참조(읽기)                                     │ 산출물
          ▼                                                 ▼
┌─────────────────────┐                          ┌─────────────────────┐
│  Review (검도)       │                          │  장비일람표/도면      │
└─────────────────────┘                          └─────────────────────┘

Shared Kernel: Capacity(용량), ModelCode(모델식별), ComboRatio(조합비), PowerSupply(전원), EnergySource(계열: EHP/GHP/AWHP...)
```

관계: 장비마스터 → (검도, 생성)은 **Customer/Supplier** (마스터가 상류). 생성·검도는 마스터의 PUBLISHED 데이터만 소비하며 역방향 의존 금지.

## 3. 구현 계획 — 도면 생성 (Generation)

**생성 메뉴의 목적은 장비를 배치해 장비선정표·장비일람표를 뽑는 것이다**(주인님 확정 2026-07-10).
도면 산출은 그 부산물이다. 모든 스텝은 이 두 산출물의 컬럼을 채우는 행위로 정당화된다.

파이프라인 (4스텝, `src/presentation/generation/steps.ts`가 정본):

```
도면 업로드(실 검출 완료) → [1 실내기 배치] → [2 실외기 선정·조합] → [3 실외기 배치] → [4 산출물 생성]
```

> **실 검출은 스텝이 아니라 초기 상태다** (주인님 확정 2026-07-16): 도면을 열면 실이 이미 검출돼 있다
> (`App.tsx`의 `seedDetectedRooms`가 로드 시점에 실·형상을 시딩). 검출 결과 다듬기 — 시설군(단위부하 전제)
> 선택, 실 자르기(V)·병합(M) — 는 첫 스텝(실내기 배치)에서 이어서 한다. (참고: `doc/03_데이터/방검출.xlsx`)

> **순서의 근거**: 실내기를 다 배치해야 총 정격용량이 확정되고, 그래야 그 부하를 감당할 실외기를 고를 수 있다.
> 몇 대가 필요한지 정해져야 어디 둘지 정한다. (선정이 배치보다 앞선다 — 2026-07-10 교체)

1. **실내기 배치**: 실별 부하(냉방 W) 계산 → 타입·대수·모델 선정 후 도면에 심볼 배치.
   - 진입 시 실은 이미 검출돼 있다. 시설군은 이 단계의 프로젝트 설정(단위부하의 전제) — 바꾸면 그 시설군의 단위부하로 실을 재시딩한다.
   - 실 형상 편집(자르기 V·병합 M)은 이 단계 전용이다(검출 결과를 다듬는다).
   - **도면 심볼 1개 = 실내기 1대 = 선정표 대수 1** (`Placement.positions.length === quantity` 불변식).
   - 도면에서 지우면 대수가 줄고 조합비가 다시 계산된다.
2. **실외기 선정·조합** (`domain/generation/selectOutdoorUnits.ts`):
   - **버킷**: 층 × 계열(EnergySource). 교차 계열 금지, 한 실외기가 여러 층에 걸치지 않는다.
   - **선정**: 조합비 허용범위(모델별 `comboRange`, 전역 기본 0.5~1.03)를 만족하는 **최소 용량** 모델. 최소 용량이 목표 조합비 100%에 가장 가깝다.
   - **분할**: 최대 연결 **대수**(`maxConnections`)·40HP·상한 초과 시 분할. 한 실의 대수는 쪼개지 않는다.
   - **냉난방**: 계열만으로는 못 가른다(같은 EHP에 절환형·냉방전용 공존) → 난방이 필요하면 `heatKw = null`을 배제.
   - UI: 하단 상태바 "조합 리포트"(읽기 전용) + "실외기 조합 매핑" **하단 도킹 패널**(드래그 배정/해제, 교체/분할/추가/삭제, 조합비·경고). 도면을 가리지 않는다.
3. **실외기 배치**: 선정된 실외기 심벌을 도면(건물 외부)에 배치. **좌표가 산출 도면에 실린다.** 이격거리(실외기 간 좌우 200 / 앞뒤 900mm) 검증.
4. **산출물 생성**: 장비선정표(CSV) · 장비일람표(xlsx) · 도면(SVG).

**스텝 가드** (`domain/generation/StepGuard.ts`): 전제가 안 갖춰지면 다음 단계로 못 간다.
CTA 버튼은 **항상 활성**이고, 클릭하면 차단 사유(왜)와 해결법(어떻게)을 팝업으로 알린다. 버튼을 숨기거나 죽이지 않는다.
과부하·빈 그룹·되돌리기 어려운 초기화(시설군 변경에 따른 실 재시딩, 실 자르기·병합)는 확인을 받고 진행한다.

상태·워커·큐(SQS) 재시도, 층고 변경에 따른 배관 길이 재계산 등은 백엔드 워커 도메인에서 처리(추후).

## 4. 구현 계획 — 관리자 장비마스터 (Equipment Master)

**분류 체계 (4단 계층)** — 장비일람표의 유형/분류를 최상위 축으로:

| 계층 | 예시 |
|------|------|
| 대분류(역할) | 실외기 / 실내기 / 환기 / 판넬 / 시공자재 / 제어·통신 |
| 중분류(제품군·계열) | 실외기: EHP·GHP·AWHP·수냉식·Chiller / 실내기: 1WAY·4WAY 카세트·덕트·스탠드·FCU |
| 시리즈 | Multi V Super 5, Multi V i, GHP Super III |
| 모델 | RPUW12BX9M, GPUW280C2S (마스터 레코드) |

**스펙 필드 전략:** 장비일람표 전 컬럼을 **필수 정규화 필드**로, 제품군 고유 항목은 **JSONB 확장 스펙**으로. (근거: `doc/02_DB설계/스펙_정규화_공통화_분석_보고서.docx`, `doc/01_기획/장비마스터_기획_분류체계_및_UI구성_v0.1.docx`)

**단일 메뉴 내부 뷰:** 목록 / 상세·등록·수정 / 엑셀 업로드·동기화 / 게시·단가. (조합 매핑은 마스터가 아니라 유저단)

**게시 게이트:** DRAFT → PUBLISHED → ARCHIVED. 검도·생성은 PUBLISHED만 조회. (스키마: `doc/02_DB설계/01_schema_v2.sql`)

## 5. 아키텍처 원칙 (필수 준수)

### 5.1 Clean Architecture

의존성은 **항상 안쪽(도메인)으로만** 향한다. 도메인은 React·DB·HTTP를 모른다.

```
src/
  domain/            # ① 엔티티, 값객체(VO), 도메인 서비스 — 프레임워크 무지, 순수 함수/클래스
    equipment/       #    장비마스터 컨텍스트
    generation/      #    생성 컨텍스트 (Room, OutdoorGroup, 배치/조합 규칙)
    review/          #    검도 컨텍스트
    shared/          #    공유 커널 (Capacity, ModelCode, ComboRatio, EnergySource)
  application/       # ② 유즈케이스(인터랙터) + 포트(인터페이스). 도메인만 의존.
  infrastructure/    # ③ 어댑터: 리포지토리 구현, 장비마스터 API 클라이언트, worker
  presentation/      # ④ React 컴포넌트·훅 (현재 App.jsx/components → 점진 이관)
```

- **규칙 1**: `domain`은 어떤 상위 레이어도 import 하지 않는다.
- **규칙 2**: `application`은 인터페이스(포트)에만 의존하고 구현(infrastructure)은 주입(DI)받는다.
- **규칙 3**: React/`presentation`은 유즈케이스를 호출할 뿐 비즈니스 규칙을 갖지 않는다.
- **현재 POC**: `src/App.jsx`, `src/components/*`, `src/data.js`는 프로토타입(presentation+목업). 로직을 `domain`으로 추출하며 리팩터한다. 첫 예시: `src/domain/shared/ComboRatio.js`.

### 5.2 DDD (Domain-Driven Design)

- **유비쿼터스 언어**를 코드·문서·대화에서 동일하게 사용한다(§9 용어집).
- **값객체(VO)**: 불변, 자기검증. 예: `Capacity`, `ModelCode`, `ComboRatio`, `PowerSupply`.
- **애그리거트**: `Product`(장비마스터), `OutdoorGroup`(실외기+연결 실내기 조합), `GenerationJob`, `ReviewJob`. 애그리거트 루트로만 상태 변경.
- **리포지토리**: 애그리거트 단위 저장/조회 인터페이스는 `application`의 포트로 선언, 구현은 `infrastructure`.
- **도메인 이벤트**: 예 `ModelPublished`, `OutdoorGroupReassigned`.
- 불변식(invariant)은 도메인 안에서 강제한다(예: 조합비 범위, 계열 호환).

### 5.3 TDD (Test-Driven Development)

**Red → Green → Refactor**를 엄격히 따른다.

- 새 규칙/기능은 **실패 테스트 먼저** 작성 후 최소 구현.
- 우선순위: 도메인 단위테스트 > 유즈케이스 테스트 > 통합 > E2E.
- 테스트 명명: `describe`(대상) + `it('~하면 ~한다')` (Given-When-Then).
- 도구: **Vitest**. 명령: `npm test`(1회), `npm run test:watch`(감시).
- 도메인/애플리케이션 레이어 커버리지 목표 **90%+**. presentation은 핵심 상호작용 위주.
- 규칙이 바뀌면 테스트부터 바꾼다.

### 5.4 적대적 QA (Adversarial QA)

구현을 **공격하는** 관점으로 검증한다. "정상 경로"만 보지 않는다.

- **경계값**: 조합비 0.49/0.5/1.3/1.31, 실내기 0개/최대치, 빈 도면, 방 1개.
- **악의적/이상 입력**: 계열 불일치 드롭, 존재하지 않는 모델, 음수 용량, 초대형 엑셀(>10MB), 깨진 DXF.
- **실패 주입**: worker 타임아웃, SQS 메시지 유실, API 5xx, 네트워크 단절.
- **동시성**: 같은 실내기를 두 그룹에 동시 배정.
- 기법: negative test, property-based/fuzz(가능 시), 회귀 테스트.
- **프로세스**: 기능 구현 후 별도 "QA 리뷰"(가능하면 서브에이전트)로 위 시나리오를 능동 탐색하고, 발견 결함은 실패 테스트로 고정 후 수정한다.

### 5.5 Claude Hooks (자동 게이트)

`.claude/settings.json`에 훅을 정의해 **품질 게이트를 자동화**한다(이미 시딩됨).

| 이벤트 | 동작 |
|--------|------|
| `PreToolUse` (Bash) | `./.claude/hooks/guard.sh` — 위험 명령(`rm -rf /`, `git push -f` 등) 차단(exit 2). |
| `PostToolUse` (Edit\|Write) | `./.claude/hooks/format.sh` — 수정 파일 자동 포맷(prettier). |
| `Stop` | `npm test --if-present` — 세션 종료 전 테스트 게이트(그린 유지). |

- 훅은 비차단·안전하게 작성한다(포맷 실패가 작업을 막지 않도록 `|| true`).
- 위험 작업(대량 삭제, 강제 푸시, 시크릿 노출)은 훅 또는 사람 확인으로 차단한다.

### 5.6 SRP (단일 책임 원칙) — 필수

> 프로젝트가 커졌다. **모든 모듈·클래스·함수는 변경 이유가 하나여야 한다.**
> "이 파일이 왜 바뀌는가?"에 대한 답이 둘 이상이면 쪼갠다.

**판별 기준 — 변경 이유(actor)로 나눈다**

한 파일이 아래 중 둘 이상을 이유로 수정된다면 SRP 위반이다.

| 변경 이유 | 소속 |
|-----------|------|
| 비즈니스 규칙이 바뀌어서 | `domain` |
| 유즈케이스 흐름이 바뀌어서 | `application` |
| 저장 방식·외부 API가 바뀌어서 | `infrastructure` |
| 화면 표현·상호작용이 바뀌어서 | `presentation` |

**적용 규칙**

- **한 파일 = 한 책임**. 파일 200줄은 상한이 아니라 *냄새 경보*다. 줄 수가 아니라 **책임 개수**로 판단한다.
- **함수는 한 가지 일만 한다**. "그리고(and)"로 이어지는 이름(`parseAndSave`, `validateAndPublish`)은 분리 신호다.
- **값객체는 자기 자신만 검증한다**. 다른 애그리거트를 알면 책임이 샌 것이다.
- **컴포넌트는 렌더링만**. 데이터 가공·비즈니스 판단은 훅이나 유즈케이스로 내린다.
- **한 테스트 = 한 동작**. 테스트가 여러 이유로 깨지면 대상이 여러 책임을 지고 있다는 뜻이다.
- **정책과 메커니즘을 섞지 않는다**. "무엇을 할지(정책)"와 "어떻게 할지(메커니즘)"를 같은 함수에 두지 않는다.

**리팩터 트리거** — 다음을 발견하면 즉시 분리한다.

- 하나의 필드/플래그가 서로 다른 세 가지를 결정한다 → 축을 분리하거나, 정말 같은 축임을 근거로 문서화한다
  (예: `is_vrf`는 ①모델명 HP 인코딩 ②maxConn 게시 요건 ③생성단 조합 후보 노출을 함께 가른다.
   실데이터에서 세 성질이 정확히 일치함을 확인하고 `doc/05_설계결정/마력_환산식_적용_검토.md §5`에 근거를 남겼다)
- 버그 하나를 고치는데 무관해 보이는 테스트가 깨진다
- 파일 상단 import가 여러 레이어에 걸쳐 있다
- 조건 분기가 "이 호출자는 A, 저 호출자는 B"를 구분한다 → 함수를 둘로 나눈다

**SRP는 파일을 잘게 쪼개라는 뜻이 아니다.** 함께 변하는 것은 함께 두고, 다른 이유로 변하는 것만 떼어낸다.

## 8. 참고 자료

모든 근거 문서는 `doc/`에 복사되어 있고 경로 인덱스는 [`doc/index.md`](doc/index.md)에 있다. 신규 기획/설계 결정은 해당 문서를 근거로 인용한다.

> ⚠️ **doc/ 자료 신뢰도 주의 (필수):** `doc/` 폴더에는 **최신화되지 않은(오래된) 자료가 섞여 있다.** 문서 간 값·정의가 상충하거나, 문서와 코드가 어긋나거나, 요구가 애매모호할 때는 **임의로 추정해서 진행하지 말고 반드시 주인님(사용자)에게 먼저 질문한다.** 특히 스키마·필드 정의·단가/등급 등 수치·비즈니스 규칙이 문서마다 다르면 어느 것을 기준으로 삼을지 확인받은 뒤 구현한다. (근거를 인용하되, 그 근거가 현행인지 확신이 없으면 확인 우선)

## 8.1 권한 (Authorization)

로그인·세션은 아직 구현하지 않았다. **권한 규칙만 먼저 세워 두고 사용자 역할을 하드코딩한다**(주인님 지시 2026-07-10).

- 규칙은 도메인에 둔다: `src/domain/auth/Permission.ts` (`Role`, `Principal`, `canManageEquipment`).
- **fail-closed**: 사용자를 알 수 없거나 모르는 역할이면 거부한다. 새 역할이 규칙에 안 적혔다면 조용히 열리는 것보다 조용히 막히는 편이 안전하다.
- 목업 사용자는 `src/data.ts`의 `CURRENT_USER.role`. 일반 사용자 화면을 보려면 `'USER'`로 바꾼다.
- 인증이 붙으면 `CURRENT_USER`를 세션에서 만든 `Principal`로 바꾸기만 한다. 규칙과 화면 분기는 그대로 남는다.

**메뉴를 숨기는 것은 권한 통제가 아니다.** 반드시 진입 경로도 막는다.
- GNB 메뉴 노출: `canManageEquipment(CURRENT_USER) && <a …>`
- 라우팅 차단: `main.tsx`가 권한 없으면 `ForbiddenPage`(403)를 렌더하고 **저장소(SQLite)를 열지도 않는다.**
- 실서비스에서는 서버가 먼저 거부한다. 클라이언트 게이트는 그 응답을 사용자에게 옮기는 자리일 뿐이다.
