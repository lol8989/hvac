# 마력(HP) 환산식 적용 검토

> 작성일 2026-07-10 · 지시: 주인님 (Confluence 환산식을 장비마스터 등록 시 적용)
> 상태: **구현 완료** — 상수 2907 · 1HP 미만 소수 보존 · `hp_source` 기록 · 계열별 게시 요건 분리 (주인님 확정 2026-07-10)

## 1. 출처

Confluence [실내기·실외기 자동배치 룰 및 배치 기준 정리](https://tosky.atlassian.net/wiki/spaces/LA/pages/1296859176) (작성자 황민태, 0708 회의 확정사항 절) — "환산식" 항목 원문:

```
1 kcal : 860kw
1hp : 2500 kcal
1hp : 2900w

kcal/h 기준
1마력 = 2500 kcal/h

W 환산
2500 kcal/h ÷ 0.86 = 2907 W
```

같은 문서 1부 ①은 부하 산정식을 명시한다:

```
필요부하(kW) = 면적(㎡) × 단위부하(kcal/h·㎡) ÷ 860
860 = 1kW를 kcal/h로 환산한 값 (장비선정표 엑셀과 동일 기준)
```

## 2. 원문의 표기 문제 (해석 확정)

| 원문 | 문제 | 정확한 표기 |
|------|------|------------|
| `1 kcal : 860kw` | 단위가 뒤집혀 있다 | **1 kW = 860 kcal/h** |
| `1hp : 2900w` ↔ `2500 ÷ 0.86 = 2907 W` | 같은 문서 안에서 두 값이 다르다 | 2500 kcal/h ÷ 0.86 = **2906.98 W** |

`860`과 `0.86`은 같은 상수의 배수 관계이므로 두 식은 정합하다. `2900W`는 반올림 표기로 판단한다.

**본 프로젝트 채택 상수** (주인님 확정 2026-07-10: **2907**):

```
HP_TO_KCAL = 2500 kcal/h        (1마력)
KCAL_TO_W  = 1 / 0.86 = 1.1628  (기존 코드의 UnitLoad ×1.163과 동일 계보)
HP_TO_W    = 2500 / 0.86 = 2906.98 W ≈ 2907 W
```

기존 코드 `src/domain/shared/UnitLoad.ts`가 이미 `kcal/h → W`에 ×1.163을 쓰고 있어 계보가 일치한다.

## 3. 실데이터 검증 (1,206모델 시드 전수)

환산식을 **HP 산출식**으로 쓸 수 있는지 실외기 670건(HP·냉방용량 동시 보유)으로 대조했다.

| 판정 기준 | 결과 |
|-----------|------|
| `round(coolingW / 2907) == HP` | 309/670 (46.1%) |
| `ceil(coolingW / 2907) == HP` | 401/658 (60.9%) |
| 가장 가까운 표준 HP 격자로 스냅 | 440/658 (66.9%) |
| 오차 ±3% 이내 | 556/670 (83.0%) |
| 오차 ±10% 이내 | 658/670 (98.2%) |

**중분류별 실측 W/HP 중앙값** — 계열마다 다르다:

| 중분류 | 모델 수 | 중앙값 (W/HP) |
|--------|--------|--------------|
| EHP 냉난방 절환형 | 444 | 2,871 |
| EHP 냉방전용 | 103 | 2,830 |
| 수냉식 | 59 | 2,900 |
| EHP 동시형 | 49 | 2,850 |
| GHP | 15 | 2,813 |

### 결론 3-1 — 환산식은 HP의 **산출식이 아니라 검증식**이다

정확한 정수 HP를 재현하지 못한다(최대 66.9%). 계열별 W/HP가 2,813~2,900으로 흩어져 단일 상수로 역산하면 GHP 계열이 체계적으로 낮게 나온다(예: `GP-W501C2S` 실제 50HP, 환산 48.9HP).

따라서 **HP의 1차 출처는 모델명 유도를 유지**한다(P3 결정 `doc/NEXT.md`, `src/domain/equipment/ModelCode.ts`). 환산식은 아래 두 용도로 쓴다.

1. **판별기(disambiguator)** — 모델명 해석이 애매할 때 정답을 고른다 (§4).
2. **백필(backfill)** — 모델명이 HP를 인코딩하지 않는 계열의 근사 HP를 채운다 (§5).

## 4. 환산식이 검출한 결함 — 100HP 이상 모델 12건 오독

### 증상

`horsepowerFromModelCode`는 접두 알파벳 뒤 **첫 두 자리**를 HP로 읽는다(`ModelCode.ts:9`, `MAX_HP = 99`). 100HP 이상 모델이 **10HP로 오독**된다.

| 모델 | 시트 냉방능력 | 현행 HP | 실제 HP | 환산 검증 |
|------|--------------|---------|---------|-----------|
| `RP-Q1001X9S` | 280.60 kW | 10 | **100** | 100 × 2907 = 290.7 kW |
| `RP-Q1021X9S` | 285.90 kW | 10 | **102** | 102 × 2907 = 296.5 kW |
| `RP-Q1041X9S` | 291.20 kW | 10 | **104** | 104 × 2830(냉전 실측) = 294.3 kW |
| `RP-W1001X9H` | 286.20 kW | 10 | **100** | 100 × 2871 = 287.1 kW |
| `RP-W1021X9H` | 294.30 kW | 10 | **102** | 102 × 2871 = 292.8 kW |
| `RP-W1041X9H` | 302.40 kW | 10 | **104** | 104 × 2907 = 302.3 kW ✅ |

각 모델의 변종(`RP-Q100PX9S`, `RP-W100BX9P` 등)을 포함해 **총 12 레코드**.

시트 원값은 정상이다 — `RP-Q1001X9S`의 `냉방능력 > 정격`은 `280.60 kW`이고 `냉방능력 > 정격 (kcal/h)`는 `241,300`으로, 241,300 ÷ 860 = 280.6 kW로 자체 정합한다. **데이터가 아니라 유도 규칙이 틀렸다.**

### 왜 모델명만으로는 못 고치는가

숫자 블록의 마지막 자리는 HP가 아니라 변형 코드인데, 그 코드가 숫자일 수도 문자일 수도 있다.

| 모델 | 숫자 블록 | 해석 |
|------|----------|------|
| `RPUW281X9P` | `281` (3자리) | HP=**28** + 변형 `1` |
| `GP-W560C2S` | `560` (3자리) | HP=**56** + 변형 `0` |
| `RP-Q1001X9S` | `1001` (4자리) | HP=**100** + 변형 `1` |
| `RP-Q100PX9S` | `100` (3자리) | HP=**100** + 변형 `P`(문자) |

`RPUW281`(3자리 → 28HP)과 `RP-Q100P`(3자리 → 100HP)가 같은 블록 길이에서 갈린다. **블록 길이만으로는 판별 불가**하며, 냉방용량 환산이 유일한 판별 근거다.

### 영향 범위

12건 전부 `status = DRAFT`이므로 생성·검도 소비 경로에는 **현재 무영향**이다(게시본만 소비). 다만 게시하는 순간 조합비가 약 10배 틀어진다(100HP 실외기를 10HP로 인식 → 조합비 과대).

### 수정 (완료)

모델명에서 HP 후보 2개(앞 2자리 / 앞 3자리)를 뽑고, 냉방용량 환산값에 더 가까운 쪽을 채택한다. 3자리 후보는 100~199 범위만 인정한다(`281`→281HP, `001`→1HP 오인식 차단). 냉방용량이 없으면 기존대로 앞 2자리.

`horsepowerFromModelCode(modelCode, coolingW?)` — 시그니처에 냉방용량을 선택 인자로 추가했다.

## 5. HP 없는 모델 백필 (주인님 지시 2026-07-10)

모델명이 HP를 인코딩하지 않는 계열은 `horsepower = null`이었다. 이 때문에 **게시 전제조건(실외기는 HP 필수)에 걸려 게시가 차단**됐다(커밋 `efc6f11`: 칠러 20건 전량 차단, DRAFT 실외기 195건 제외).

| 중분류 | HP 없음 | 냉방용량 보유 (백필 가능) |
|--------|--------:|------------------------:|
| 시스템 에어컨(단품) | 84 | 71 |
| 냉장·냉동 CDU | 53 | 53 |
| 공랭식 스크롤 칠러 | 42 | 42 |
| 수냉식 스크롤 칠러 | 11 | 11 |
| AWHP | 5 | 2 |
| **합계** | **195** | **179** |

이 계열들은 모델명 숫자가 **용량**을 뜻한다(`LSC-V1200C9` → 12.0 kW, `TUW1002A2SR` → 10.0 kW). 모델명 유도를 적용하면 12HP·10HP로 오독하므로, 환산식 백필이 **유일한 수단**이다.

```
HP = round(coolingW / 2906.98)          1HP 이상
HP = round2(coolingW / 2906.98)         1HP 미만 — 소수 보존 (주인님 지시)
```

1HP 미만은 `LSC-G0100F2`(1kW = 0.34HP) 1건뿐이다. 1HP로 클램프하면 3배 과대 표기되므로 소수를 보존한다. `products.horsepower`는 `REAL`이라 그대로 담긴다.

### VRF 판별 축 — `is_vrf`

백필은 계열을 가려서 적용해야 한다. 그런데 **`energy_source`로는 못 가른다**: `수냉식`에 VRF(Multi V Water)와 칠러가 함께 있고, `EHP`에 Multi V와 단품(SINGLE)이 함께 있다.

기존 빌드 스크립트에만 있던 `derivesHp` 플래그를 `is_vrf`로 이름을 바꿔 `product_series` 테이블로 올렸다. 실데이터에서 **세 성질이 정확히 일치**하기 때문이다.

| `is_vrf` | 모델명이 HP를 인코딩하는가 | 게시에 최대 연결 실내기 수를 요구하는가 | 생성단 조합 후보로 노출되는가 |
|---------|:---:|:---:|:---:|
| true (24 시리즈) | ✅ | ✅ | ✅ |
| false | ❌ (숫자는 용량) | ❌ (개념 없음) | ❌ |

### 게시 전제조건의 계열별 분리 (필수 동반 변경)

**HP만 백필해서는 게시가 한 건도 풀리지 않았다.** 179건 전부 `max_connections`가 없고, 게시 게이트가 실외기에 그 값을 요구했기 때문이다.

'최대 연결 실내기 수'는 VRF(실외기 1대 ↔ 실내기 N대) 계통의 개념이다. 칠러(냉수 배관)·CDU(쇼케이스)·단품(1:1)에는 존재하지 않는다. 따라서 비-VRF 실외기는 이 요건을 면제한다(주인님 확정).

대신 **비-VRF는 게시되더라도 생성단 조합 후보로 노출하지 않는다** — `SqliteEquipmentMaster.publishedOutdoor()`가 `is_vrf = 1`만 조회한다. `OutdoorSpecFields.maxConn`이 `number` 계약이라, 이들이 새어 들어가면 `NaN`이 된다.

### 마력 출처 (`hp_source`)

백필된 HP는 실측이 아닌 추정치다(오차 ±10% 내 98%, 정수 일치 67%). 출처를 데이터에 남긴다.

| 값 | 의미 | 시드 건수 |
|----|------|---------:|
| `MODEL_CODE` | 모델명 유도 (VRF) | 667 |
| `DERIVED` | 냉방용량 환산 추정 | 179 |
| `CURATED` | 큐레이션 시드값 | 3 |
| `MANUAL` | 관리자 직접 입력 | — |

관리 목록의 HP 컬럼은 `DERIVED`를 `22 (추정)`처럼 구분 표기한다.

### 잔여

실외기 16건은 냉방용량 자체가 없어 백필할 수 없다(AWHP 3, 단품 13). 게시는 `냉방 용량이 없어 게시할 수 없습니다`로 계속 차단된다.

| 항목 | 결정 |

| 채택 상수 | **2907 W/HP** (= 2500 ÷ 0.86). 원문의 `2900w`는 반올림 표기로 간주 |
| 백필 반올림 | **반올림**(1HP 이상), 1HP 미만은 소수 둘째 자리 보존 |
| 추정 HP의 게시 | **허용** + `hp_source` 기록. 비-VRF는 생성단 조합 후보에서 제외 |
| 계열별 상수 | **미도입** — 단일 2907 유지(Confluence 합의값). §3 실측표는 검증용으로만 보존 |

## 6-1. 남은 과제

- 스펙시트에 마력 항목이 있는 냉장·냉동 CDU 3건은 시트값을 그대로 쓰는 편이 정확하다(현재는 환산 백필). `hp_source = 'SHEET'` 도입 시 처리.
- 백필 HP가 Confluence의 "단일 실외기 최대 40HP" 분할 한도에 걸리는 비-VRF 모델이 34건 있다(칠러 45~193HP). 칠러는 VRF 분할 대상이 아니므로 현재는 무해하나, 생성단이 칠러를 다루게 되면 재검토가 필요하다.
