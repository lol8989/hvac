## 3. 선결 과제 — 목업 실을 인접 배치로 교체

## 4. 병합(Merge)의 규칙

### 4.1 인접 판정 — 공유 변이 있어야 한다

두 폴리곤이 **양의 길이를 갖는 공유 경계**를 가질 때만 병합한다. 꼭짓점 하나만 닿는 경우(대각선 접촉)는 병합하지 않는다 — 합쳐도 이어진 실이 아니다.

```
공유 변 O          공유 변 X (점 접촉)
┌────┬────┐        ┌────┐
│ A  │ B  │        │ A  │
└────┴────┘        └────┼────┐
                        │ B  │
                        └────┘
```

### 4.2 합집합 — 경계 순회

내부가 겹치지 않고 변을 공유하는 두 단순 폴리곤의 합집합은 **경계 순회**로 구한다(외부 라이브러리 없이).
1. 두 폴리곤을 같은 방향(CCW)으로 정규화한다. 공유 변은 서로 **반대 방향**으로 나타난다.
2. 겹치는 공선 구간의 끝점에서 변을 쪼갠다(부분 겹침 대응).
3. 공유 구간을 뺀 나머지 변만 남기고, 시작점–끝점을 이어 하나의 닫힌 고리로 만든다.
4. 결과가 하나의 고리가 아니면(구멍·분리) 병합을 거부한다.

결과는 **오목할 수 있다**(ㄴ자·ㅁ자). 그래서 E4가 필요하다.

### 4.3 병합된 실의 값

| 항목 | 규칙 |
|------|------|
| **면적** | `area(A) + area(B)` — 정확히 합. (자르기의 "면적 합 보존" 불변식과 대칭) |
| **단변·장변** | 합집합 폴리곤의 **OBB**에서 재계산. ⚠️ 오목 실의 OBB는 볼록 껍질 기준이라 과대평가된다 — 한계로 문서화하고, 타입 결정이 뒤집히면 사용자가 선정표에서 조정 |
| **용도** | **면적이 큰 쪽**을 승계(E2). 용도가 다르면 확인 팝업이 부하 변화를 숫자로 보여준다 |
| **층** | 두 실의 층이 다르면 **병합 불가**(실외기 버킷이 층 × 계열이다) |
| **id·이름** | E1. 형제(`AC_001-1`,`AC_001-2`)면 부모(`AC_001`, `거실`) 복원. 아니면 앞선 id + `거실+침실1` |
| **위치(행 순서)** | 앞선 실의 자리에 삽입하고 뒤의 실은 제거 — 선정표 행·층 섹션 순서 유지 |
| **단위부하 오버라이드** | 용도를 승계한 쪽(큰 실)의 오버라이드를 물려받는다 |

### 4.4 실내기(배치)

자르기와 대칭이다 — **도면 심볼이 진실**이다.
- 두 실의 심볼 좌표를 **그대로 합친다**. 대수 = 합친 심볼 수.
- 모델이 서로 다르면(한 실 = 한 모델이 원칙) **대수가 많은 쪽**의 모델을 승계한다(동수면 면적이 큰 쪽). 확인 팝업에 명시하고, 'AI 실내기 배치'로 다시 뽑을 수 있다.
- 유닛 id는 새 실 id로 다시 부여된다(`AC_002#1` → `AC_001#2`). 좌표는 그대로라 도면은 변하지 않는다.

### 4.5 실외기 조합

두 실이 **서로 다른 실외기 그룹**에 배정돼 있으면, 합친 실은 한 실이면서 두 그룹에 걸리게 된다 → `AssignmentPlan`의 **실 응집 불변식** 위반.
→ 병합 시 두 실의 배정을 **모두 풀어 미배정 풀로 보낸다**. 확인 팝업에 명시하고, 기존 가드(`UNASSIGNED_ROOMS`)가 다음 단계를 막는다.

## 5. 자르기의 오목 폴리곤 일반화 (E4)

지금 `Polygon.splitByLine`은 Sutherland–Hodgman 반평면 클리핑이라 **볼록 전용**이다(오목이면 잘못된 조각이 나온다 — 그래서 명시적으로 throw하고 있다).

교체할 알고리즘 — **교점 다리(bridge) 연결**:
1. 폴리곤 정점을 순회하며 절단선과의 교점을 경계에 삽입한다.
2. 교점을 선 방향 파라미터 `t`로 정렬한다. 짝수 개가 나오고, 연속 쌍 `(t0,t1) (t2,t3) …` 중 **폴리곤 내부를 지나는 구간**만 다리가 된다(중점이 내부인지로 판정).
3. 경계 + 다리로 이루어진 그래프를 순회해 조각을 만든다.
4. 볼록 폴리곤이면 결과는 정확히 2조각 — 기존 결과와 동일해야 한다(회귀 테스트).

`sliceRoom`은 **조각이 2개일 때만** 자른다. 3개 이상이면 `SliceProducesManyPiecesError`로 거부하고 "이 각도로는 실이 3조각으로 나뉩니다 — 위치나 각도를 바꿔 보세요"라고 알린다(실 하나 → 둘이라는 규칙을 지킨다).

## 6. Undo / Redo (E3)

### 6.1 편집 상태를 하나로 묶는다 — `World`

지금 편집 상태가 5개의 `useState`로 흩어져 있다(`domainRooms` / `roomGeom` / `placements` / `outdoorPositions` / `plan`+repo, 그리고 `facility`). 히스토리는 **원자적 스냅샷**이어야 하므로 하나로 묶는다.

```ts
interface World {
  rooms: Record<string, DomainRoom>     // 실(부하·용도·면적)
  geom: Record<string, Polygon>         // 실 형상(베이스 좌표)
  placements: Record<string, Placement> // 실내기 배치(대수·모델·좌표)
  outdoorPositions: Record<string, Pt>  // 실외기 심볼 좌표
  plan: AssignmentPlan                  // 실외기 조합·배정
  facility: FacilityType                // 단위부하의 전제
}
```

**히스토리에 넣지 않는 것**: `step`(파이프라인 단계), `selRooms`(선택), 패널 폭·모드 등 UI 상태. 되돌리기는 *편집*을 되돌리는 것이지 *보기*를 되돌리는 게 아니다.

### 6.2 `useUndoable<World>`

```ts
{ past: World[], present: World, future: World[] }
commit(mutator, label)  // past.push(present) → present 교체 → future 비움 (최대 50, 초과 시 오래된 것부터)
replace(mutator)        // 히스토리 없이 present만 교체 — 파생 동기화(syncPlanUnits) 전용
undo() / redo()
```

- **파생 동기화가 히스토리를 오염시키지 않게** 한다: `placements`가 바뀔 때 도는 `syncPlanUnits` 효과는 `replace`를 쓴다. 안 그러면 Ctrl+Z 한 번에 "플랜 동기화"만 되돌아가고 정작 사용자의 편집은 그대로 남는다.
- `InMemoryPlanRepository`는 `present.plan`을 따라간다 — undo/redo 후 `repo.save(present.plan)`.
- undo 후 **사라진 실이 `selRooms`에 남으면 파생값이 undefined를 받아 터진다** → undo/redo 직후 선택을 `present.rooms` 기준으로 필터한다(자르기에서 이미 겪은 함정).

### 6.3 UI

- **Ctrl+Z** 되돌리기 / **Ctrl+Shift+Z**·**Ctrl+Y** 다시하기. 입력 필드에 포커스가 있으면 무시(뷰어의 `typing()` 가드를 `presentation/keyboard.ts`로 추출해 공유 — SRP).
- WorkBar에 **↶ / ↷** 버튼. 되돌릴 게 없으면 비활성. `title`에 무엇이 되돌아가는지 라벨을 보여준다("실 자르기 되돌리기").
- 되돌린 뒤 토스트: "실 자르기를 되돌렸습니다".

## 7. 뷰어 M(병합) 모드

- 하단 플로팅 도구 메뉴에 **실 병합 / M** 항목 추가(자르기 바로 아래).
- **2단계 선택**: 첫 클릭 = 실 A(굵은 테두리로 고정 강조) → 커서를 올리면 **인접한 실만** 강조된다(인접하지 않으면 흐리게 + 커서 옆 "인접 아님"). 두 번째 클릭 = 실 B → 병합.
- **Esc** 또는 우클릭 → A 선택 해제(모드는 유지).
- HUD: "실 병합 · 붙어 있는 실을 두 번 클릭하세요" → A 선택 후 "거실 + ? · 인접한 실을 클릭".
- 자르기와 같이 **검출 단계 전용**. 다른 단계에서 M을 누르면 안내 토스트.

## 8. 구현 순서 (TDD — Red 후보)

### N0. 목업 실을 인접 배치로 교체 (§3)
- **Red**: `data.test.ts` — "6실은 빈틈없이 인접한다"(모든 실이 최소 한 실과 양의 길이 공유 변을 갖는다 / 두 실의 내부가 겹치지 않는다).
- **Red(회귀)**: 실내기 타입·대수·조합비·실외기 선정 결과가 바뀌지 않는다(`App.report.test.tsx` 그대로 통과).
- `App.slice.test.tsx`의 클릭 좌표를 새 박스로 갱신.

- **Red**: 나란한 두 사각형을 합치면 정점 4개의 사각형(공유 변이 사라진다) / 넓이 = 합 / ㄴ자 배치를 합치면 정점 6개 오목 폴리곤 / 부분 겹침 변(한쪽이 더 긴 벽)도 옳게 합쳐진다 / 점만 닿으면 `NotAdjacentError` / 떨어져 있으면 `NotAdjacentError`.

### N2. `splitByLine` 오목 일반화 (E4)
- **Red(회귀)**: 볼록 폴리곤의 절단 결과가 **기존과 동일**(정점 수·넓이·순서).

- **Red**: `sliceRoom`이 3조각이면 `SliceProducesManyPiecesError`.

### N3. `mergeRooms` 도메인 서비스 + `guardMerge`
- **Red**: 면적 = 합 / 층이 다르면 거부 / 용도가 다르면 큰 실의 용도를 승계하고 **부하가 단순 합이 아니다**(그 사실을 검증) / 형제면 부모 id·이름 복원 / 남남이면 앞선 id + 합성 이름 / 단변·장변이 합집합 OBB에서 나온다 / 인접하지 않으면 거부.
- **Red**: `guardMerge` — 배치·조합이 없으면 ALLOW, 있으면 CONFIRM(무엇이 풀리는지·부하가 얼마나 바뀌는지).

### N4. `World` 통합 + `useUndoable`
- **Red**: `useUndoable` 단위 — commit이 past를 쌓고 future를 비운다 / replace는 히스토리를 남기지 않는다 / 50개를 넘으면 오래된 것부터 버린다 / undo→redo가 원상복구.
- App 리팩터: 5개 `useState` → `world` + 접근자. **행동 변화 없음**(기존 1013개 테스트가 그대로 통과해야 한다 — 이게 리팩터의 안전망).

### N5. Undo/Redo 배선

### N6. 뷰어 M 모드
- **Red**: M을 누르면 도구바가 '실 병합' / 첫 클릭 후 인접한 실만 강조된다 / 인접하지 않은 실을 클릭하면 `onRoomsMerge`가 불리지 않고 안내가 뜬다 / **[회귀] 에어컨 모드의 M은 아무 일도 하지 않는다**(현재 M은 미사용 키다) / 검출 단계가 아니면 모드로 들어가지 않는다.

## 9. 새로 생기거나 흔들리는 불변식

1. **면적 합 보존(병합)**: `area(merged) === area(A) + area(B)` — 자르기와 대칭.
2. **부하 합은 보존되지 않는다**(용도가 다르면). 이것은 결함이 아니라 규칙이다 — 확인 팝업이 사실을 알린다.
3. **실 응집**: 병합된 실은 한 그룹에만 속해야 한다 → 배정을 푼다.
4. **유닛 id 재부여**: `${roomId}#${n}`. 좌표는 불변, id만 바뀐다.
5. **히스토리 무결성**: 파생 동기화는 히스토리를 남기지 않는다. 남기면 Ctrl+Z가 사용자가 한 적 없는 일을 되돌린다.
6. **폴리곤 단순성**: 합집합 결과가 자기교차하거나 구멍을 만들면 병합을 거부한다.
