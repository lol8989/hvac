## 0. 한 줄 요약

실명(usage)은 **단위부하를 정하는 유일한 키**다. 실명이 표에 걸리지 않으면 부하가 기본값
150kcal/h·㎡로 떨어지고, **화면상으로는 정상값과 구분되지 않는다.** 추출기가 붙기 전에
"얼마나 새는지"를 볼 수 있게 만들어 두고, 그 실측으로 동의어 사전을 짓는다.

---

## 1. 왜 이 문서가 필요한가

### 1.1 실명이 부하를 정한다

```
실명(usage) + 시설군(facility) + 부하강도 → 단위부하(kcal/h·㎡)
        → 필요부하(W) = 단위부하 × 면적
        → 실내기 타입·대수·모델 → 실외기 조합비 → 장비일람표
```

실명 하나가 최종 장비 발주까지 곧장 이어진다. **실명이 틀리면 장비가 틀린다.**

### 1.2 지금은 틀려도 조용하다

`lookupUnitLoadKcal()`은 숫자만 돌려준다. 그래서 호출부가 이 둘을 구분할 수 없었다.

| 상황 | 반환 |
|------|------|
| 관리실 — 표에 있고 standard가 150 | `150` |
| 처치실 — 표에 **없어서** 폴백 | `150` |

목업 6실로 실측한 결과(시설군 OFFICE)다:

| 실명 | 표 등재 | 단위부하 |
|------|---------|----------|
| 회의실 · 사무실 | O | 150 (근거 있음) |
| **거실 · 침실1 · 로비 · 탕비실** | **X** | **150 (폴백)** |

**6실 중 4실이 근거 없이 150이었고, 화면에는 아무 표시도 없었다.**
실무 도면(병원·숙박·산업)은 실명 종류가 훨씬 많아 이 비율이 더 나빠진다.

### 1.3 표의 크기

원본 PDF를 코드가 1:1로 완전히 옮겨놨다(요약본이 아니다). 그런데 **전체 43개 실명**뿐이다.

| 시설군 | 등재 실명 | 실무 도면에서 흔한데 빠진 예 |
|--------|-----------|------------------------------|
| 개인병원 | 병실·수술실·물리치료실·원장실·로비·방사선실 (6) | 진료실·처치실·대기실·접수·약제실·간호사실 |
| 숙박시설 | 객실·로비·커피숍·세미나실 (4) | 프런트·라운지·연회장·피트니스 |
| OFFICE | 사무실·임원실·회의실·휴게실·관리실·업무시설·근생 (7) | 탕비실·서버실·창고·복도·화장실 |

동의어 사전(`ALIASES`)은 현재 **6개 항목**뿐이다(강당·세미나·세미나실·미술실·교실·창고).

> **문제의 본질은 "실명 충돌"이 아니라 "커버리지"다.** 두 시설군에 걸치는 실명은
> 식당·로비 2개뿐이지만, 표에 아예 없는 실명은 훨씬 많다.

## 2. 이번에 구현한 것 — 폴백 관측

사전을 지어 두고 데이터를 기다리는 대신, **폴백을 관측 가능하게 만들어 실데이터가 사전을 짓게 한다.**

### 2.1 조회에 근거를 붙였다

```ts
resolveUnitLoadKcal(facility, usage, intensity)
  → { kcal: number, matched: UsageMatch, resolvedUsage: string }
```

| `matched` | 뜻 | 조치 |
|-----------|-----|------|
| `exact` | 실명이 표에 그대로 있다 | — |
| `normalized` | 표기 정규화 후 걸렸다 (`회의실-2` → `회의실`) | — |
| `alias` | 동의어로 흡수됐다 (`강당` → `회의실`) | — |
| `unknown` | **실명은 있는데 표에 없다** | **동의어 사전에 추가할 후보** |
| `unnamed` | **실명 자체가 없다** | **도면에 실명을 기입해야 한다** |

`unknown`과 `unnamed`는 값이 똑같이 150이지만 **조치가 다르다.** 전자는 우리가 사전을
고치면 되고, 후자는 도면이 고쳐져야 한다. 그래서 반드시 구분해서 넘겨받아야 한다(→ §3.2).

### 2.2 표기 정규화 (도메인 추정 없음)

실명을 **추정하지 않는다.** 표기 규칙만 적용한다.

| 규칙 | 예 |
|------|-----|
| 괄호 주석 제거 | `사무실(A동)`, `사무실 [3층]` → `사무실` |
| 공백 접기 | `회 의 실` → `회의실` |
| 번호 접미사 제거 | `회의실1`, `회의실-2`, `회의실 03` → `회의실` |

번호 제거는 **남는 게 있을 때만** 한다. `101`처럼 숫자뿐인 실명을 빈 문자열로 만들면
`unnamed`와 구분이 사라지기 때문이다.

### 2.3 화면 표시

장비선정표(`⧉ 선정표 확인`)의 실명 셀에 배지가 붙는다.

- `표에 없음` (unknown) — 실명 확인 또는 단위부하 직접 입력
- `실명 없음` (unnamed) — 도면에 실명 기입 필요

근거가 있는 실(`exact`/`normalized`/`alias`)에는 배지를 달지 않는다 — 모든 행에 붙으면
정보가 되지 못한다.

## 3. 개발팀 확인 요청 (3건)

이음매를 정해야 어느 쪽이 무엇을 맡을지 확정된다. **아래 3건에 답을 주시면 반영한다.**

### Q1. 추출기 출력은 날 텍스트입니까, 정규화된 실명입니까?

- **날 텍스트**라면 §2.2 정규화를 우리 쪽(`normalizeUsage`)이 계속 맡는다
- **정규화해서 준다면** 규칙을 공유해 달라 — 양쪽이 각자 정규화하면 규칙이 어긋난다.
  그 경우 우리 쪽 정규화는 제거하고 `exact`만 남긴다

> 우리 쪽 정규화 규칙은 §2.2가 전부다. 추가/삭제 의견을 주면 반영한다.
> (예: `제1회의실` 같은 **선행** 서수는 현재 처리하지 않는다 — 필요하면 알려 달라)

### Q2. 이름을 못 찾은 실을 구분해서 넘겨줍니까?

`unnamed`와 `unknown`을 가르려면 추출기가 "라벨 없음"을 명시해야 한다.

- 빈 문자열 `""` / `null` / `"미상"` / `"미지정"` / `"-"` 는 **무명으로 해석**한다
  (`UNNAMED_USAGE = '(무명)'`, `isUnnamed()`)
- 다른 표기를 쓸 계획이면 알려 달라 — 토큰 목록에 추가한다

> Confluence 한계 항목에 *"도면상 라벨을 밖으로 빼서 표시하는 경우 실면적 산출과
> 방 라벨 매칭에서 혼선 발생"* 이 적혀 있다. 이 케이스가 `unnamed`로 오는지,
> 아니면 엉뚱한 실명으로 붙어 오는지가 중요하다. 후자면 화면에서 잡을 방법이 없다.

### Q3. 한 실에 라벨 후보가 여럿일 때 하나로 확정해서 줍니까, 후보 목록으로 줍니까?

- **확정해서 준다면** 지금 계약(`usage: string`) 그대로면 된다
- **후보 목록이라면** 우리가 표에 걸리는 후보를 고를 수 있다(정확도가 올라간다).
  이 경우 계약을 `usage: string | string[]`로 넓힌다 — **답을 받고 나서** 바꾼다

## 4. 다음 단계 (답을 받은 뒤)

1. **실측 수집** — 추출기 출력으로 프로젝트를 돌려 `unknown` 실명과 빈도를 뽑는다
2. **사전 확정** — 그 목록을 현업이 검토해 `실명 → 표 행` 매핑을 확정한다.
   **개발이 임의로 매핑하지 않는다** (단위부하는 장비 용량 직결 수치 — `CLAUDE.md §8`)
3. **시설군별 사전으로 확장** — 현재 `ALIASES`는 전역 맵이지만, 흡수 대상이 시설군마다
   다르다(병원 `대기실`→로비 / 호텔 `대기실`→로비, 그러나 병원 `사무실`→원장실).
   `Record<FacilityType, Record<string,string>>` + 전역 폴백 층으로 넓힌다

> **시설군 축은 유지한다.** 실명 우선 조회(실명으로 시설군을 역추론)는 검토했다가 폐기했다 —
> 로비·사무실·회의실·식당·창고 같은 범용 실명이 모든 시설군에 나타나 병원 로비를
> 종교시설 값으로 끌어올 수 있다. 시설군이 제 값을 하는 자리는 충돌 2건이 아니라
> **미등록 실명을 어느 행으로 흡수할지 정하는 것**이다.
