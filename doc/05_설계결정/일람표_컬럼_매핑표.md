## 0. 현재 상태

지금 나오는 `장비일람표.csv`는 컬럼이 5개뿐이다: `구분, 모델명, 사양, 수량, 연결`.
`사양`은 개별 필드가 아니라 미리 조립된 설명 문자열(`"냉난방 절환형 · 냉방 34.8kW · EHP"`)이고,
데이터 소스도 도메인이 아니라 목업 `ModelCard`다 (`src/presentation/generation/schedule.ts`).

반면 롱테일 스펙은 이미 DB에 다 있다 — P4에서 1,206모델을 `product_specs.spec_data`(JSONB)에 통째로 적재했다.
**문제는 그것을 읽는 경로가 없다는 것이다.** `SqliteEquipmentMaster`는 `product_specs`를 `SELECT`하지 않는다(INSERT 경로만 존재).

## 4. 넘어야 할 세 가지

### (1) 라벨 변종 — 계열마다 키가 다르다

같은 의미인데 라벨이 갈린다. 실측(1,206모델 전수):

| 의미 | 실내기 | 실외기 | 변종 수 |
|------|--------|--------|--------|
| 냉방능력 | `냉방능력 > 공칭용량` | `냉방능력 > 정격` | 21 |
| 전원 | `전원 > Case 1` | `전원 > Case 1`, `전원` | 17 |
| 배관경 | `냉매 연결 배관경 > 액관` | `배관경 > 냉매 액관` | 55 |
| 소음 | `실내기 소음(음압 레벨) > …` | `실외기 소음(음압 레벨) > 냉방 / 난방` | 34 |

→ **`spec_label_aliases`(제품군별 라벨 → 표준 키)가 사실상 선행 조건이다.**
`doc/02_DB설계/02_seed_v2.sql`에 샘플 28건이 있으나 키 형식이 `"섹션.라벨.단위"` 3단이고,
실제 적재된 키는 `"섹션 > 라벨"` 2단이라 그대로 맞물리지 않는다.

### (2) 값 파싱 — 저장은 원문, 출력은 정형

`Φ6.35 (1/4)` → `6.35`, `32 / 25` → `32`, `220, 1상(2선), 60` → `1, 2, 220, 60`.
**어디서 변환할 것인가**가 결정 사항이다.

- **(a) 적재 시점 정규화**: 업로드 때 표준 키 + 정형 값으로 저장. 출력이 단순해지지만 원문 손실 위험.
- **(b) 출력 시점 변환**: 원문을 보존하고 일람표를 만들 때 변환. 원문이 SSOT로 남지만 변환기가 계열별로 복잡.

### (3) 시트에 없는 컬럼

`반입치수`는 스펙시트 어디에도 없다. `누전차단기 수량`은 판독본에서도 `-`다.

## 5. 구현 순서 (제안)

1. **읽기 경로 신설** — `product_specs` 조회 포트(`EquipmentSpecRepository.specsOf(modelCodes)`).
   현재 `SqliteEquipmentMaster`는 `product_specs`를 SELECT하지 않는다.
2. **라벨 정규화 사전** — `spec_label_aliases` 테이블 + 실측 라벨 기준 시드(2단 키 형식).
3. **값 변환기** — `presentation/equipment/scheduleFormat.ts` (숫자 추출·전원 재배열·치수 정리).
4. **일람표 빌더 교체** — `buildScheduleRows`를 도메인 + 스펙 조인 기반으로. 계열별 컬럼 집합 분리.
5. **xlsx 출력** — 현재 CSV+BOM. 컬럼이 24~39개라 병합 헤더가 필요하면 xlsx가 사실상 필수.

## 6. 미확정 — 주인님 확인 필요

1. **값 정규화 위치**: 적재 시점(a) vs 출력 시점(b).
2. **시트에 없는 컬럼**(반입치수 등): `-`로 채울 것인가, 컬럼 자체를 뺄 것인가.
3. **계열별 시트 분리**: 실내기/EHP 실외기/GHP 실외기를 별도 시트로 낼 것인가(정본은 GHP 전용 39컬럼).
4. **출력 포맷**: CSV 유지 vs xlsx 전환(병합 헤더 2행 재현 여부).
