## 1. 목적 · 배경

장비마스터(Equipment Master)는 장비 모델·스펙·단가의 **단일 진실 공급원(SSOT)** 이다. 현재 POC는
스펙을 하드코딩 시드로 들고 있었고(→ `refactor(equipment) 5b0e0f3` 로 마스터 컨텍스트로 이미 이관·게시게이트 도입 완료),
이제 그 마스터를 **SQLite 저장소**로 실체화하고 **관리 페이지**를 붙인다.

**데이터 흐름 (핵심 원칙)**

```
[입력] 현업 담당자가 업로드              [마스터: SSOT]                 [출력/산출물]
LG 스펙시트(~40종 xlsx, 제품군별)  ──ETL──▶  장비마스터(SQLite)  ──참조(PUBLISHED)──▶  장비선정표 · 장비일람표
  · 전치형(모델=열, 항목=행)                 · 4단 분류 + 정규화 + JSONB           · 실사용 필드는 "일부(hot)"뿐
  · 제품군마다 항목 상이                     · 게시게이트 DRAFT→PUBLISHED→ARCHIVED
```

> **주인님 지침(2026-07-09):** "현업은 스펙시트 형태로 업로드하고, **선정표와 일람표에만 마스터 데이터가 대부분 쓰인다**."
> → 구조 설계의 축: **입력(스펙시트)의 풍부함은 다 담되(JSONB 롱테일 포함), 출력(선정표/일람표)이 쓰는 필드는 정규화 컬럼(hot)으로 1급 취급.**

## 3. 마스터 구조 설계 (schema_v2.sql 기반, SQLite 이식)

`doc/02_DB설계/01_schema_v2.sql`(PostgreSQL)이 이미 이 흐름을 정확히 모델링한다. POC는 그 **핵심 서브셋**을 SQLite로 이식한다.

### 3.1 채택 테이블 (POC 1차 서브셋)

```
product_categories        4단-① 대분류(실외기/실내기/환기/판넬/제어…)
product_subcategories     4단-② 중분류·계열(EHP/GHP/AWHP/수냉식/Chiller · 1WAY/4WAY/덕트…) + energy_source
product_series            4단-③ 시리즈(Multi V Super 5, GHP Super III…) + mfl_code
products                  4단-④ 모델(SSOT 레코드) + 정규화 hot 필드 + status(게시게이트)
product_specs             JSONB 롱테일(제품군 고유 항목)
product_prices            단가(유형별·기간별, 현행가=effective_end_date NULL)
price_types               단가 유형(소비자가/공급가…)
efficiency_grades / energy_sources / refrigerant_types / power_supplies   룩업
```

2차 이후(§7 Phase 3+): `import_jobs` / `import_job_items` / `import_reject_log`(스펙시트 ETL), `spec_label_aliases`(라벨 정규화), `product_combinations` / `indoor_outdoor_compat`(조합·호환).

### 3.2 정규화 hot 필드 vs JSONB 분류 원칙

> **선정표·일람표가 읽는 필드 = 정규화 컬럼(products), 그 외 스펙시트 롱테일 = product_specs.spec_data(JSON).**

`products` 정규화 컬럼(출력이 직접 소비):
`model_code, name_display, product_series_id, refrigerant_type_id, power_supply_id, efficiency_grade_id,
horsepower, cooling_capacity_w, heating_capacity_w, heating_capacity_cold_w(한랭지),
power_consumption_cool_w, power_consumption_heat_w, cop_cooling, cop_heating,
weight_kg, dim_width_mm/height/depth, manufacturer, status, published_at, discontinued_at`

`product_specs.spec_data`(JSON) 예: 송풍기 풍량/정압, 배관구경(액/가스/드레인), 반입치수, 전선 규격, 누전차단기, 운전전류, 연료소비량, 압축기/오일, 소음 등 **일람표 세부·제품군 고유 항목**.

### 3.3 PostgreSQL → SQLite 이식 규칙

| PG | SQLite(POC) |
|----|-------------|
| `BIGSERIAL PRIMARY KEY` | `INTEGER PRIMARY KEY AUTOINCREMENT` |
| `TIMESTAMPTZ` / `DATE` | `TEXT`(ISO8601) |
| `JSONB` | `TEXT`(JSON 문자열, `json_extract`로 조회) |
| `BOOLEAN` | `INTEGER`(0/1) |
| `NUMERIC(p,s)` | `REAL` 또는 `INTEGER`(원 단위) |
| `INET[]` / 배열 | 제외(POC 불필요) |
| 트리거(updated_at/published_at) | 앱 레이어에서 처리 |
| 부분 인덱스 / GIN | 일반 인덱스로 단순화 |
| 뷰 `v_published_*` | SQLite `VIEW` 또는 리포지토리 쿼리로 재현 |

### 3.4 게시 게이트

`products.status ∈ {DRAFT, PUBLISHED, ARCHIVED}`. 생성·검도는 **PUBLISHED만** 조회(현 `EquipmentMaster.publishedIndoor/Outdoor` 계약 유지). `v_published_products` 상당의 뷰/쿼리로 노출.

### 3.5 조합비 정책 = UI 설정값 (주인님 결정 2026-07-09)

> 선정표 "100% 이내" vs 현 POC "0.5~1.3"의 상충은 **코드 하드코딩이 아니라 관리 UI에서 설정**해 해소한다.

- **모델**: 조합비 상·하한(`comboMin`/`comboMax`)을 **정책값**으로 저장.
  - **전역 기본**: `system_settings`(key=`combo.ratio.min`/`combo.ratio.max`) — 예 하한 0.5 / 상한 1.0(=100%) 또는 1.3.
  - **제품군별 override**: `product_subcategories`(또는 별도 `combo_policies`)에 계열/중분류별 `combo_min`/`combo_max` — DOAS 등 예외 흡수.
  - 우선순위: 제품군별 override > 전역 기본.
- **도메인**: 이미 존재하는 `ComboRange`(VO, min·max 자기검증) + `OutdoorUnit.comboRange`가 이 값을 운반한다. 마스터 스펙(`OutdoorModelSpec.comboRange`)에 정책값을 주입 → 생성단 조합비 경고/판정이 설정값을 따른다.
- **UI**(관리 페이지, §5): 「조합비 정책」 설정 화면 — 전역 min·max 입력 + 제품군별 표(행=중분류/계열, 열=min·max), 저장 시 마스터에 반영. 변경은 즉시 생성단(선정표 조합비 경고선)에 전파.
- **경고 vs 차단**: 조합비 범위는 **경고**(배정 거부 아님) 규칙 유지(`OutdoorGroup` 주석대로) — 설정값은 경고 임계만 바꾼다.
- **표기**: 내부 저장은 비율(0.5, 1.0, 1.3). UI 표기는 % 병기 가능(50% / 100% / 130%).

## 4. 아키텍처 배치 (Clean Architecture)

- **domain/equipment** (기존): `PublishStatus`, `MasterRecord`, `EquipmentMaster`(읽기 포트). → 유지·확장.
- **application/equipment** (신규): 쓰기 포트 `EquipmentMasterRepository`(register/update/publish/archive/setPrice…), 유즈케이스.
- **infrastructure/equipment/sqlite** (신규):
  - `sqlSchema.ts`(DDL), `sqlSeed.ts`(초기 시드 — 현 16+7을 products/series/…로 정규화), `SqliteDb.ts`(sql.js 초기화 + IndexedDB 로드/저장),
  - `SqliteEquipmentMaster.ts`(읽기 포트 구현 — PUBLISHED 스냅샷 materialize), `SqliteEquipmentMasterRepository.ts`(쓰기).
- **동기/비동기 경계**: sql.js 초기화는 비동기 → **부트스트랩(`main.tsx`)에서 await 후** PUBLISHED 스냅샷을 동기 배열로 materialize하여 `EquipmentMaster`(동기 포트)로 제공. App은 `master` 옵셔널 prop(기본=인메모리 → 테스트 무영향)로 주입.
- **영속**: 변경 시 `db.export()` → IndexedDB 저장, 부팅 시 복원. 없으면 시드로 초기화.
- **포트 유지 이점**: 생성/검도는 코드 변경 없이 마스터 백엔드만 교체. 추후 실서버 API로도 포트만 교체.

## 7.1 P1 상세 작업 목록 (착수 시)

1. `sql.js` 의존성 추가(WASM). Vite에서 `?url`/public로 wasm 로드, Vitest(node)에서도 동작 확인.
2. `sqlSchema.ts`: §3.1 서브셋 DDL(SQLite). `products.status` 체크, 현행가 인덱스.
3. `sqlSeed.ts`: 현 실내기 16(4WAY/덕트)·실외기 7을 categories/subcategories/series/products/prices로 정규화 시드 + 게이트 실증용 DRAFT/ARCHIVED.
4. `SqliteDb.ts`: `initSqlJs` + IndexedDB 로드/저장(브라우저), 인메모리 모드(테스트).
5. `SqliteEquipmentMaster.ts`: 부팅 시 PUBLISHED 조회 → `publishedIndoor/Outdoor` 동기 배열 materialize(현 계약 그대로).
6. `main.tsx`: `await createSqliteEquipmentMaster()` → `<App master=… />`. 실패 시 인메모리 폴백.
7. `App`: `master` 옵셔널 prop(기본=`defaultEquipmentMaster`). 기존 테스트 무영향.
8. 테스트: 스키마/시드/게이트(인메모리 sql.js), App 회귀, (수동) 새로고침 영속.
